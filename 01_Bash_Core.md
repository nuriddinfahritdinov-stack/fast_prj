# **Инструментарий Автоматизации**

## **1\. Введение и Статус**

* **Статус:** Базовый модуль.  
* **Назначение:** Этот файл — фундамент. Все последующие справочники по PostgreSQL, Oracle и MongoDB опираются на механику работы со средой, потоками и переменными, описанную здесь.  
* **Требование:** Выполнять в среде bash (не sh и не zsh для обеспечения кросс-платформенности скриптов).

## ---

**2\. Переменные Среды (Environment)**

Для DBA переменные — это не просто хранилища данных, это способ сказать скрипту, к какому инстансу подключаться.

### **Ключевые принципы:**

* **Локальные переменные:** DB\_PORT=5432 — видны только в текущем скрипте.  
* **Экспорт:** export PGDATA="/var/lib/postgresql/data" — делает переменную доступной для всех дочерних процессов (например, для самой утилиты psql).

### **Разбор синтаксиса:**

Bash

\# Плохо: скрипт упадет, если переменная не задана  
\# Хорошо: использование значения по умолчанию (Default Values)  
USER\_NAME=${DB\_USER:-"guest"}   
\# Если $DB\_USER пуста, выберется "guest". Это защищает скрипт от пустоты.

## ---

**3\. Потоки данных и Перенаправления**

Работа с СУБД — это всегда передача текста. Ты либо отправляешь SQL в базу, либо забираешь логи из неё.

### **Стандартные дескрипторы:**

1. **stdin (0):** Ввод (откуда приходят команды).  
2. **stdout (1):** Вывод (результат запроса).  
3. **stderr (2):** Ошибки (сообщения о падении коннекта).

### **Примеры для практики:**

* **Слияние потоков:** pg\_dump my\_db \> backup.sql 2\>&1*Почему это важно?* Если бэкап упадет, текст ошибки попадет в тот же файл backup.sql, и ты увидишь причину при просмотре файла, а не потеряешь её в консоли.  
* **Пайпы (Pipes):** cat script.sql | psql \-d mydbМы «пробрасываем» содержимое файла прямо в "рот" CLI-интерфейсу базы данных.

## ---

**4\. Here-Documents (Инъекция кода)**

Это лучший способ писать SQL-запросы прямо внутри Bash-скрипта, сохраняя их читаемость.

Bash

\#\!/bin/bash  
\# Переменная внутри Bash  
TARGET\_SCHEMA="audit\_logs"

\# Начинаем блок ввода до метки EOF  
sqlplus \-s / as sysdba \<\<EOF  
SET HEADING OFF;  
SET FEEDBACK OFF;  
\-- Bash подставит значение переменной прямо в SQL:  
SELECT count(\*) FROM ${TARGET\_SCHEMA}.events WHERE status \= 'ERROR';  
EXIT;  
EOF

* **Лайфхак:** Если написать \<\<'EOF' (в кавычках), Bash не будет пытаться подставлять свои переменные внутрь блока, что полезно, если в SQL есть свои символы $.

## ---

**5\. Проверка состояния (Exit Codes)**

В автоматизации нельзя просто «надеяться», что команда выполнилась. Нужно проверять переменную $?.

* 0 — Команда выполнена успешно.  
* 1-255 — Произошла ошибка.

### **Пример логики «Защищенного скрипта»:**

Bash

\# Выполняем проверку целостности  
mongo \--eval "db.adminCommand('checkMetadata')"

if \[ $? \-ne 0 \]; then  
    echo "\[$(date)\] ERROR: Проверка метаданных MongoDB провалена\!" \>\> /var/log/db\_monitor.log  
    exit 1  
fi

## ---

**6\. Обработка результатов (Command Substitution)**

Иногда нужно сохранить результат SQL-запроса в переменную Bash, чтобы принять решение дальше.

Bash

\# Записываем текущее количество соединений в переменную  
CONN\_COUNT=$(psql \-t \-A \-c "SELECT count(\*) FROM pg\_stat\_activity;")

if \[ "$CONN\_COUNT" \-gt 100 \]; then  
    echo "Внимание: Высокая нагрузка\! Соединений: $CONN\_COUNT"  
fi

* \-t \-A — флаги PostgreSQL, которые убирают рамки таблиц и заголовки, оставляя только «чистое» число.

## ---

**7\. Безопасность и Best Practices**

* **set \-e**: Добавь это в начало скрипта. Если любая команда вернет ошибку, скрипт немедленно остановится, не пытаясь выполнить следующие шаги (например, не начнет удалять логи, если бэкап не создался).  
* **set \-u**: Выдаст ошибку, если ты используешь переменную, которую забыл объявить (защита от опечаток вроде $DB\_NMAE).  
* **Абсолютные пути**: Всегда пиши /usr/bin/psql вместо просто psql в cron-задачах, так как у cron ограниченный PATH.
